What it is

A small HTTP service that lets CI run a test against real hardware and get structured pass/fail + logs back — without the calling repo hosting a self-hosted runner.

The controller is the seam between four parties. A caller (a GitHub Actions job, or any client) submits a job and waits for the result on a long-poll. The controller host runs this repository: it owns the HTTP API, the HTMX dashboard, the SQLite job store and the scheduler, and it has no devices attached to it. The HIL host fleet is a set of bench Raspberry Pis the controller fans work out to over SSH; each host owns a slice of the devices under test. The ProtoMQ broker is what firmware talks to as an MQTT client during a test. It is per-session, not a fixed host: the controller can launch it on itself (tachyon, 192.168.1.169), on a DUT host (e.g. rpi-hil004), or point the firmware at any other reachable broker — a staging server such as io.adafruit.us, or live io.adafruit.com. The controller writes the chosen host:port into the DUT's secrets at flash time and observes the broker rather than mandating one.

The defining choice is that the work happens on the HIL host, not the controller. Each worker holds an SSH session to the host that owns its assigned device and drives flashing, power control, serial capture and camera grabs there. The controller only orchestrates. One process, one SQLite file, an asyncio worker pool — no Redis, no Celery, no external job broker.

Architecture

One orchestrator, a fleet of bench hosts, a read-only-by-default dashboard over the same API.

A thin HTMX dashboard sits in front of the same /v1 API for human inspection of queue state, device availability, the wiring graph and per-job logs. Read-only pages are public on the LAN; every write — submitting a job, editing topology, toggling power — requires a bearer token or a verified GitHub Actions OIDC token. The topology endpoints are also available to CI, so a job can ask "is there a seat for me?" via POST /v1/topology/resolve before submitting, instead of finding out by failing mid-flash.

Concurrency model: per-device asyncio.Lock (a job holds its device until terminal) plus a per-host semaphore — SBC hosts default to one job at a time, microcontroller hosts are bounded by their per-device locks. An optional exclusive-host lock lets a hard-to-trace job take a whole host alone and capture unambiguous dmesg / usbmon.

Services, ports, keys & endpoints

Where each service listens, what guards every hop, and why the network layer is interchangeable.

The network seam is swappable

The controller is just an HTTP service on :8080. CI reaches it however the network allows — over Tailscale MagicDNS today (…ostrich-escalator.ts.net:8080), but a plain LAN address (192.168.1.169:8080), a WAN host (wan.gdenu.fi:8080), or any other VPN works identically. Only two CI inputs change with the path: HIL_API_BASE (where the controller is) and the network-join step (TAILSCALE_AUTHKEY, or whatever a different VPN needs). The thing that actually authorises a job — a bearer token or GitHub OIDC — is independent of the transport that carried it.

USB power plane (the solenoid): "toggle USB" is not a software port-disable — it's a physical solenoid pulse. The Genesys USB hub's per-port power is driven by an MCP23017 GPIO expander at I²C 0x20 (bus 1); usb_hub.py / solenoid_hub_control.py pulse the channel mapped to each device, and a per-board timing profile (standard, samd51_uf2, …) shapes the power/reset pulse so the board lands in the right bootloader. After a power-cycle, usbipd (:3240) re-attaches the DUT. Manual equivalents: turn_on.sh / turn_off.sh / solenoid_hub_cli.py.

The first topology.yaml is seeded from two existing artefacts: hardware.md (power/reset wiring — the 8-channel solenoid map, USB VID:PID, timing profiles) and the ProtoMQ scripts/*.json demos (device↔display wiring + display interface). An importer cross-references the two and flags anything unresolved; humans own the final file.

Setup & deployment

Three tiers, installed via the Python package plus deploy scripts.

1 · Controller host

Any Linux box (Pi, NUC, VM) with network reach to the bench. uvicorn hil_controller.main:app behind Caddy/nginx for TLS. SQLite (WAL) + artifacts under /var/lib/hil/. YAML config in /etc/hil/, hot-reloaded. Per-host SSH keys in /etc/hil/keys/ (mode 0400).

2 · HIL host fleet

scripts/setup-hil-host.sh authorises the controller key, installs usbip-autobind, masks ModemManager, and (on Pi Zero) opts into the dwc2 driver. On rpi-displays it installs the solenoid hub control + udev rules for the MCP23017 and stable serial symlinks.

3 · ProtoMQ broker

Per-session, not a dedicated box. The controller launches a ProtoMQ broker on itself (tachyon), on a DUT host (e.g. rpi-hil004), or points the DUT at an external broker — a staging server (io.adafruit.us) or live io.adafruit.com (MQTT 8883). Bench ProtoMQ uses MQTT 1884 + web UI 5173; the host:port is filled into secrets.json at flash time, never a CI input.

Caller side (CI)

A downstream repo copies the examples/hil-call.sh submit + long-poll flow and sets a handful of secrets. The broker host:port is not a CI input — the controller's write_secrets_msc stage fills io_url/io_port from the per-session broker it launches.

Process flow — the regression suite

A GitHub PR drives a real-hardware A/B comparison: low_ref = a published release, high_ref = this PR's build.

After the build workflow produces per-target artifacts, hil-test-suite.yml joins Tailscale, asks GET /v1/targets which boards are live, downloads the release and PR firmware, uploads it via POST /v1/firmware, then submits a test array — each test is its own driver script reported individually. Tests run if: always() so one failure doesn't hide the others, and the workflow posts a fresh PR comment per run (not sticky) with a per-target table and inline serial-log excerpts.

Test 1 — device check-in (the default gate)

hil-checkin-run.sh: flash the PR build → write secrets → power-cycle → verify_checkin. Passes when the firmware connects to the broker and announces itself — verdict CHECKIN_VERDICT ok=true. This proves the board boots and provisions before any deeper test runs.

Tests 2 & 3 — pixelWrite regression (#926, fixed by #927)

hil-pixelwrite-run.sh runs the same firmware-bench pipeline twice (low vs high). A v1 pixelWrite to an uninitialised NeoPixel strand crashes release beta.127 (rebooted=true) but is handled gracefully by the fix (rebooted=false). The harness compares the two builds; the stage itself doesn't pass/fail.

Test 4 — display proof (spec-driven)

hil_display_test.py + a JSON spec: flash → v2 check-in → inject_protobuf display Add/Write → capture_display. The test sequence is data, so any board/display (or a non-display flow — e.g. an I²C sensor add between writes) is a new spec, not new code. Verdict DISPLAY_CAPTURE_VERDICT; a cropped, white-balanced camera image is embedded in the PR comment alongside the time-aligned serial/protomq proof.

Inside firmware-bench (per job)

enter_bootloader→ erase→ flash→ verify→ launch_protomq→ write_secrets_msc→ power_cycle→ inject_pixelwrite→ verdict

Signal injection is the crux: ws_signal_inject encodes the exact 11-byte nanopb PixelsWriteRequest, connects to the per-session broker, waits for pinConfigComplete, then fires the write via ProtoMQ's POST /api/echo. It then watches MQTT for a fresh re-checkin — a reboot inside the window means the firmware crashed; silence means it survived. Stages are pluggable (STAGE_HANDLERS), so a different regression is just a different stage list + assertion — no orchestrator edit.

Generic injection & visual proof. inject_protobuf is the general stage: it publishes any ws.signal.BrokerToDevice — a builder kind the controller encodes from params (e.g. display_add_i8080) or a raw payload_hex — so display Adds/Writes and component/I²C adds reuse the same /api/echo path. verify_checkin is v2-aware, watching both the v1 (…/wprsnpr/#) and v2 (…/ws-d2b/#) check-in topics. For displays, capture_display then photographs the lit panel — it locks the autofocus-converged dioptre (continuous AF drifts mid-grab and blurs text), grabs a manual-exposure still bright enough for a self-lit TFT, crops the device ROI (scaled frame-relative to the native ?full=1 frame), and white-patch white-balances off the lit text — emitting DISPLAY_CAPTURE_VERDICT + a JPEG asset. A display-proof job is just a different stage list: … → verify_checkin → inject_protobuf (display Add) → inject_protobuf (display Write) → capture_display.

Job lifecycle

Every transition writes an append-only event; the same event log powers the long-poll wakeup, the live HTMX log view and the audit trail.

queued→ assigned→ preparing→ flashing→ running→ finished error timeout cancelled

finished carries a result of pass / fail — the device ran the test and the test gave a verdict. That is deliberately distinct from error (infrastructure failure: flash failed, USB-IP gone) and timeout (a phase budget was exceeded). CI consumes this over GET /v1/jobs/{id}/wait?since=…, re-issuing on each timeout (300s default, 600s ceiling) until the job is terminal, then pulls the serial.log / protomq.log / flash.log assets.

Manual operation

Every automated action is also a hands-on control — for bring-up, debugging a wedged board, or just eyeballing a display.

From the dashboard (or the CLI helpers — solenoid_hub_cli.py, turn_on.sh/turn_off.sh) an operator can log in to a device's owning host and open a serial console, toggle USB power and reset per solenoid channel, flash or deploy on demand through the same adapters a job uses, drive the cameras (snapshot, set ROI, QR-assisted calibration, autofocus + manual exposure/gain and focus-lock for bright self-lit panels, and the capture_display cropped white-balanced proof), and submit ad-hoc jobs with the permissive raw-firmware-smoke / git-clone-and-run built-ins behind the trusted-firmware gate. Feedback streams straight back: live serial tail over HTMX SSE, camera frames and ROI snapshots, and the captured flash/protomq logs — plus dmesg/usbmon when a job holds the host exclusively.

Extending to Wippersnapper Python

The same job-submission shape and the same controller spine — only the payload kind and the deploy/runner adapter change.

The Arduino firmware path uses a firmware-binary payload flashed onto a microcontroller. The Python single-board-computer client (currently private/unreleased) uses a git-source payload instead: the GitDeploy adapter clones the ref with submodules onto an SBC host (rpi-hil00N) — or onto the controller itself via LocalTransport for pytest runs with no real hardware — renders the chosen secret profile into .env/secrets.json, runs an optional pip install -e ., then hands off to tests/runner.py, which shells pytest -m <marker> on the target host and streams the report back as the job result.

Everything else is reused unchanged: auth and policy, the scheduler and host transport, availability and the /v1/targets matrix, secret-profile materialisation, log/camera capture, the long-poll and the per-run PR comment. The existing hil-detection pytest conftest — which already SSHes into a HIL host, toggles the USB hub, flashes and mounts CIRCUITPY — is the working prototype this architecture formalises. The Python client speaks the same ProtoMQ to whichever broker the secret profile selects, so adding it is essentially a new payload kind plus a deploy/runner adapter.

Net: Wippersnapper Python testing drops in as an additional payload + adapter on top of a platform that already handles auth, scheduling, secrets, capture and reporting — and the same platform doubles as a manual bench for interactive work.
